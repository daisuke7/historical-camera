## 1. 2026-07-02 [T4] permission_handler(iOS/SwiftPM)はマニフェスト評価キャッシュに注意

- 該当箇所: 02 §5.1(permission_handler 採用)、05 §5(Info.plist キー)
- 事象: permission_handler_apple 9.4.10(SwiftPM 統合)は Package.swift のマニフェスト評価時に
  アプリの Info.plist を読み、`NSCameraUsageDescription` が無いとカメラハンドラを
  コンパイル対象から外す(request が即 denied を返し、ダイアログが出ない)。
  評価結果は `~/Library/Developer/Xcode/DerivedData` と
  `~/Library/Caches/org.swift.swiftpm/manifests` にキャッシュされるため、
  **キー追加前に一度でもビルドした環境では、キーを追加しても denied のまま**になる。
- 実装での扱い: Info.plist に利用目的キーを追加(05 §5 どおり)+ 上記 2 キャッシュを削除して
  再ビルドで解消。キー追加後に初めて評価される環境(新規クローン・CI)では発生しない。
- P0 後の反映案: 05 §5 に「permission_handler のカメラハンドラ有効化は Info.plist キーの
  存在に依存する。キーより後からビルド環境に入った場合は DerivedData と
  ~/Library/Caches/org.swift.swiftpm/manifests の削除が必要」の注記を追加。

## 2. 2026-07-02 [T6] iOS シェーダーの実測 GPU 時間

- 該当箇所: 01 §1(フィルタ処理時間 2〜6ms @720p)、08 T6(<8ms 基準)
- 事象: iPhone 12 Pro(A14)実機の XCTest 計測で、720p 最悪ケースが
  写真時代(1845 相当・blur+halation+全ノイズ)5.31ms、
  絵巻時代(Sobel+ポスタライズ+紙)7.43ms。予算 8ms 内だが、
  絵巻時代は 01 の見積り上限(6ms)を上回った。
- 実装での扱い: 予算内のため実装変更なし。GPU 計測は RunnerTests の
  testEraFilterGpuBudgetAt720p として常設(回帰検知可能)。
- P0 後の反映案: 01 §1 の見積りを「2〜7.5ms @720p(絵巻時代が最重量)」に更新。
  下位 GPU 機での再計測を P1 チューニング時に実施。

## 3. 2026-07-02 [T8] Android の回転モデルはドキュメントと異なる実挙動(重要)

- 該当箇所: 02 §4.1(バッファはセンサー向き固定)、06 §3.1(transformMatrix は回転を
  含まない)、06 §3.3(TransformationInfo で quarterTurns を算出)
- 事象(Pixel 6 で実測、3 ビルドで切り分け):
  1. SurfaceTexture の getTransformMatrix には **HAL が設定するセンサー回転(背面 90°)が
     常に焼き込まれている**(targetRotation に依らず一定)。ドキュメントの
     「回転は含まれない」は誤り。
  2. その結果、サンプル後のコンテンツは「デバイス自然向き(縦)で正立」になり、
     センサー形状(横長)のバッファに入れると縦が潰れる。
  3. TransformationInfo.rotationDegrees はメタデータに過ぎず、バッファ実回転とは無関係。
- 実装での扱い:
  - 出力バッファ(SurfaceProducer)を**回転後の寸法(例: 960x1280 縦)で確保**し、
    previewWidth/Height もその値を返す(「バッファ寸法固定」は維持)。
  - SurfaceTexture の defaultBufferSize はカメラ要求解像度(1280x960)のまま。
  - Dart の RotatedBox 用 quarterTurns は **(-displayRotation/90) mod 4**
    (コンテンツが自然向き正立のため、ディスプレイ回転の打ち消しのみ)。
    DisplayManager.DisplayListener で追従。preview.targetRotation はセンサー向きに固定。
  - 副次修正: エンジン切断後のフレーム押し込みで FlutterJNI クラッシュ
    → released フラグ+onDetachedFromEngine での dispose を追加。
  - CameraX の解像度交渉で 1280x720 要求に対し **1280x960(4:3)** が選ばれた
    (request.resolution を正とする規約どおり動作)。
- 検証: Pixel 6 実機で縦・左横・右横すべて向き/アスペクト正常、UI 追従
  (自動回転オン時)。自動回転オフ時は UI 固定+映像は常に正しい(自然な挙動)。
- P0 後の反映案: 02 §4.1 と 06 §3.1/§3.3 の回転モデル記述を上記の実挙動に全面改訂。
  iOS(05)は設計どおりで変更不要。デバイス依存の可能性(HAL が transform を
  設定しない機種)に注意書きを追加。

## 4. 2026-07-08 [T16] レンズ切替+ズームの UI 未規定部分の補完

- 該当箇所: 04 §1(レイアウト — レンズ切替ボタンの記載なし)、
  04 §5(CameraState のフィールド一覧に lens / zoom がない)
- 事象: T16 のレンズ切替+ズーム(02 §3.1 の switchLens / setZoom)には
  UI 状態としての現在レンズ・ズーム倍率と、切替ボタンの配置が必要だが、
  04 に規定がない。
- 実装での扱い:
  - CameraState に `lens`("back"/"front"、既定 back)と `zoom`
    (1.0〜、既定 1.0)を追加。
  - レンズ切替ボタンはシャッターボタンの直下(P2 まで非表示の
    モード切替セグメントの位置)に配置。
  - ズームはプレビュー領域のピンチで操作(オーバーレイ UI の背面レイヤー。
    UI 上限 8.0 倍、ネイティブ側で端末の maxZoom に再クランプ)。
  - switchLens はネイティブのコントローラ再構築で実現(textureId は変わる
    — 02 §3.1 の許容どおり)。切替・再初期化後のズームは 1.0 に戻る。
- P1 後の反映案: 04 §1 にレンズ切替ボタンの配置を、04 §5 のフィールド一覧に
  lens / zoom を追記。ズーム UI 上限(8.0)は T21 の実機チューニングで確定。

## 5. 2026-07-08 [T18] メディアフレーム演出の帯境界解釈と実装配置

- 該当箇所: 04 §7(year 帯の表)、02 §5(ファイル構成に記載なし)
- 事象: 04 §7 の帯(1930〜1970 / 1845〜1890 / 1500〜1840 / 1000〜1500)は
  連続 year での区間だが、判定粒度・境界の丸めが未規定。また実装ファイルの
  配置が 02 §5 のツリーにない。
- 実装での扱い:
  - 判定は **10 年量子化後の年**(quantizedYear)で行う。1845 開始の帯は
    量子化により 1850 の decade から適用(1840 は版画帯)。1890〜1930 と
    1970〜現在はフレームなし(表のとおり)。「1950 年代のギザフチ」は
    quantizedYear == 1950 のみ。
  - 実装は `ui/media_frame.dart`(純粋関数 `frameStyleForYear` +
    `MediaFrameOverlay`)。クロスフェードは AnimatedSwitcher の 300ms。
  - フレームの色・太さ等の見た目定数はコード内定数(§6.5 のチューニング
    対象値と同扱いで git 管理、T21 で必要なら調整)。
- P1 後の反映案: 04 §7 に量子化年での帯判定と 02 §5 のツリーへの
  `ui/media_frame.dart` 追記。
