# 01. 実現可能性の考察

## 0. 結論サマリ

| 機能 | 実現可能性 | 方式 | フェーズ |

段階的な代替アプローチを採る(§3)。

## 1. 最重要の技術判断: プレビューフィルタをどこで処理するか

「カメラ映像に毎フレーム GPU フィルタを掛けて表示する」がこのアプリの心臓部。
Flutter でカメラ映像を加工表示する方式は 4 案あり、比較検討した。

### 案A: `camera` プラグイン + `startImageStream` で Dart 処理 — ❌ 不採用

CameraImage (YUV420) が Dart 側に毎フレームコピーされる。1080p で約 3MB/frame、
30fps で 90MB/s のメモリ帯域を Dart isolate が処理することになり、
ピクセル演算を Dart(CPU)で行うと 1 フレームあたり数百 ms かかる。**実時間処理は不可能。**

### 案B: `camera` プラグインの Texture に Flutter 側でシェーダーを掛ける — ❌ 不採用

Flutter 3.7+ の `FragmentProgram`(カスタムシェーダー)は **子 widget や外部テクスチャを
サンプルできない**。`ImageFiltered`/`BackdropFilter` に自作シェーダーを適用する
`ImageFilter.shader` は Impeller 限定かつ外部テクスチャ(カメラプレビュー)との併用は
動作保証がない。粒子や傷のオーバーレイ程度は重ねられるが、セピア化・彩度変換など
**ピクセル値自体の変換ができない**ため要件を満たせない。

### 案C: PlatformView でネイティブビューを埋め込む — △ 次点

ネイティブでプレビュー+フィルタ+表示まで完結させ、Flutter に埋め込む方式。
動作は確実だが、PlatformView は合成コストが高く(特に Android の Virtual Display /
Hybrid Composition)、Flutter UI とのZ順序・タッチイベントの取り回しに難がある。

### 案D: ネイティブでキャプチャ+フィルタ処理し、Flutter の外部 Texture に流す — ✅ 採用

- ネイティブ側: カメラキャプチャ → GPU フィルタ → 加工済みフレームを
  Flutter の `TextureRegistry` に登録した外部テクスチャへ書き込む
- Flutter 側: `Texture(textureId:)` widget で全画面表示。UI は普通の Flutter widget を重ねるだけ
- フレームデータは GPU 内で完結し、Dart へのピクセルコピーは発生しない
- iOS: `FlutterTexture` プロトコル(CVPixelBuffer を返す) / Android: `SurfaceTextureEntry`

これはネイティブカメラアプリと同等のパイプラインであり、**「パフォーマンス最重要機能のみ
ネイティブ実装可」という制約に正確に合致する**。ネイティブに置くのは
「カメラ・フィルタ・保存・録画」のみで、UI・状態管理・年代→パラメータ変換ロジックは
すべて Dart に置く(§2)。

### 性能見積り(案D)

| 項目 | 見積り | 根拠 |
|------|--------|------|
| プレビュー解像度 | 1280×720(既定)。1920×1080 は 1080p 実測ゲート(§1.1)を通過した端末のみ(P1) | 720p 実測からの外挿は禁止。A14 実測で絵巻時代 7.43ms @720p → 1080p は画素数比 2.25 倍で約 16ms 相当となり、「720p で 8ms 以内」は 1080p 動作可否の根拠にならない(P0 実測により判明) |
| フィルタ処理時間 | **実測 0.7〜7.5 ms @720p 級**(絵巻時代が最重量。iPhone 12 Pro: 5.3〜7.4ms / Pixel 6: 0.7〜5.6ms)。1080p は画素数比外挿で 2〜17 ms 見込み(A14 級でも絵巻時代は予算超過があり得る → §1.1 の実測ゲートで判定) | Flutter 合成・(P2)録画パスと GPU 予算を分け合うため、合計を 33ms の 60%(20ms)以内に収める。下位 GPU 機の再計測は P1 で実施 |
| 目標フレームレート | 30fps(維持必須)、余裕があれば 60fps | |
| 発熱・電池 | サーマル監視を P0 から実装(02 §6.1) | 屋外連続利用ではスロットリングが現実的に起きる。`serious` 以上で 24fps・720p へ自動降格。非表示時は必ずセッション停止 |

### 1.1 1080p 解禁ゲート(P1。端末上でのオフスクリーンベンチ実測)

1080p の解禁可否は**その端末上で 1080p を実際に描画して測る**。判定方式は次のとおり:

- **ベンチ内容**: 1920×1080 のオフスクリーンターゲットに対し、**最悪ケースの年代パラメータ
  (絵巻時代。P0 実測で最重量 — 例: `paramsForYear(1100)` 相当)**で 30 フレーム描画し、
  1 フレームあたりの GPU 時間を実測する。判定は平均ではなく **p90 < 8ms**(瞬間的な
  ばらつきで誤判定しない)。入力は静止画パス同様の `sampler2D`/`MTLTexture` でよい
  (カメラ実映像は不要。ダミーテクスチャで GPU 負荷は等価)。
- **計測 API**: iOS は `MTLCommandBuffer` の `gpuStartTime`/`gpuEndTime`。
  Android は `EXT_disjoint_timer_query`(ES 3.0)を使い、未対応機では
  `glFinish` 挟み込みの CPU 計測で代用(保守的側に倒れるため誤解禁しない)。
- **実行タイミング**: 初回起動のプレビュー開始後、アイドル時に一度だけバックグラウンドで実行
  (初回起動をブロックしない)。結果は端末に永続化し、**次回起動から** 1080p を解禁する。
- **キャッシュ無効化**: 永続化キーにアプリバージョンを含める。シェーダーが変わり得る
  アップデート後は自動的に再計測となる。
- **解禁後の安全弁**: 1080p 動作はサーマル自動降格(02 §6.1)の対象であり、
  `serious` 以上で 720p へ降格する(ゲート通過は「常時 1080p の保証」ではない)。

## 2. 責務分割: Dart とネイティブの境界

**「年代→見た目」の知識はすべて Dart に置き、ネイティブは無知なパラメータ適用器にする。**

- Dart: スライダー値 → 西暦年 → `FilterParams`(20 個の float。02 §2 で確定。増減は 02 の改訂を伴う場合のみ)を算出(純粋関数、単体テスト可能)
- ネイティブ: 受け取った最新の `FilterParams` を毎フレーム GPU シェーダーの uniform として適用するだけ

この分割により:
1. 年代表現のチューニング(一番試行錯誤が多い部分)が Dart のテーブル 1 ファイルの修正で済み、iOS/Android 両対応が自動的に保たれる
2. ネイティブ実装は「固定シェーダー+uniform 更新」だけの単純な作りになり、低レベル LLM でも実装しやすい
3. MethodChannel の通信量は float×20 ≒ 160 byte/回。スライダードラッグ中でも 60Hz 送信で全く問題ない

## 3. 建物・人物の変化 — 何が可能で何が不可能か

### 3.1 リアルタイム生成変換(建物を別の形に描き換える)— 不可能

映像の内容を意味的に描き換える(ビル→江戸の町屋、Tシャツ→着物)には
img2img 拡散モデル等の生成 AI が必要。2026 年時点のオンデバイス推論は
最速クラスの端末でも 1 フレーム 1 秒以上(SDXL-Turbo 系の蒸留モデルでも 30fps は桁違いに届かない)。
クラウド往復でもレイテンシ・帯域・コストから動画レートは非現実的。
**「プレビューで建物の形が変わる」は初回リリースの要件から明確に外す。**

### 3.2 実時間でできる「それらしい」代替表現 — 可能(P3)

セマンティックセグメンテーションは実時間で動く:

| 対象 | 技術 | 実時間性 |
|------|------|---------|
| 人物マスク | iOS: Vision `VNGeneratePersonSegmentationRequest` / Android: ML Kit Selfie Segmentation | 30fps 可(品質 balanced 設定) |
| 空・建物・道路など | DeepLabv3 系のモバイルモデル (Core ML / TFLite) | 10〜20fps。プレビューより低頻度で更新しマスクを再利用 |

マスクを使った領域別エフェクトで「時代が変わった感」を強化する:
- 空: 色調を時代写真の空(白飛び・セピア)に寄せる
- 建物領域: コントラスト低下+テクスチャ劣化を他領域より強く掛け「風化」を演出
- 人物: 輪郭にハレーション(古い写真の滲み)を付加
- ※ 形状は変えない。あくまで「その時代に撮影された写真に写っている」という方向の演出

### 3.3 撮影後のクラウド生成 AI 変換「タイムトラベル現像」— 可能(P3)

シャッターで撮った静止画を、年代をプロンプト化してクラウドの img2img API に送り、
建物・衣装まで描き換えた 1 枚を数秒〜十数秒で返す。
- プレビュー=雰囲気(色調)のリアルタイム体験、保存後=本物の変換、という 2 段構え
- 静止画 1 枚なのでコスト・レイテンシとも現実的
- プライバシー(人物写真の外部送信)の同意 UI が必須
- 初回リリースには含めない。API 選定・コスト試算は P3 開始時に実施

### 3.4 「写真がない時代」の表現(重要な設計上の発見)

写真の発明は 1826〜1839 年。**1830 年代より前は「古い写真」という表現自体が成立しない。**
そこで年代フィルタを 3 世代に分ける(詳細は 03):

| 年代 | 表現 |
|------|------|
| 1840〜現在 | 写真の劣化表現(カラー→モノクロ→セピア、粒子・傷・ビネット、整色乾板の分光特性) |
| 1500〜1840 | 版画風(エッジ抽出+クロスハッチング+紙テクスチャ)。線密度を年代で変え、銅版の細密線(1800年代)→木版の太い線(1500年代)と遷移させる。※主市場が日本のため既定は木版・浮世絵寄りのチューニングとし、インク色・紙色の地域プリセット(和/洋)を P1 で検討 |
| 1000〜1500 | 絵巻・古文書風(墨の輪郭+和紙テクスチャ+退色顔料調) ※地域設定で洋風(フレスコ・写本風)も検討 |

これらはすべて手続き的シェーダーで実時間実装可能であり、生成 AI を必要としない。
「遡るほど記録メディアそのものが変わる」という演出はアプリのコンセプトを強化する。

## 4. 録画の実現可能性(詳細は 07)

結論: **実現可能。確立された手法が両 OS にある。**

- iOS: フィルタ済み CVPixelBuffer を `AVAssetWriter` に毎フレーム追記。マイク音声は
  `AVCaptureAudioDataOutput` から同 writer へ。録画中の年代操作は「毎フレーム最新の
  FilterParams を適用してから書き込む」だけなので追加コストなし。
- Android: `MediaRecorder`(VideoSource.SURFACE)の入力 Surface に対し、プレビューと同じ
  GL フィルタ描画をもう 1 パス実行する(1 フレームを 2 つのサーフェスに描く)。音声は
  MediaRecorder が内包処理。
- リスク: 長時間録画時の熱、音声同期、録画中の画面回転。→ 初回リリースから外し P2 とする
  (仕様上も「初回未実装可」)。設計だけ 07 で固めておき、プレビューパイプラインは
  録画分岐を後付けできる構造にしておく(02 の設計に織り込み済み)。

## 5. その他の実現可能性論点

### 5.1 縦横対応
- Flutter UI 側は `OrientationBuilder` で再レイアウトする(04 参照)
- カメラ側は「バッファはセンサー向き固定・回転は Dart 側の `RotatedBox` と EXIF で表現」
  という回転モデルに統一する(02 §4.1 で定義)。回転処理は両 OS とも API の挙動に
  罠が多い(バッファは物理回転されない等)ため、実装は必ず 02 §4.1 と 05/06 の
  該当セクションに従うこと

### 5.2 スライダー 10 年刻み × 1000 年までの操作性
- 2026→1000 年は 10 年刻みで約 103 ステップ。線形だと直近 100 年(見た目変化が最も濃い区間)が
  トラックの 1 割しかなく操作しづらい。**非線形(区分線形)スケールを採用**(04 で関数定義)。
  仕様の「10 年単位」は選択値の量子化として維持する

### 5.3 権限・ストア審査
- カメラ・マイク(録画時)・写真保存の権限。iOS は Info.plist の usage description 必須
- 実在人物の写真加工アプリとして特筆すべき審査リスクなし(P3 のクラウド送信時は要プライバシーポリシー)

### 5.4 ビルド環境の注意
- 開発・Android ビルドは Android Studio で完結する
- **iOS のビルドは Flutter が内部で Xcode ツールチェーンを呼ぶため、Mac + Xcode のインストールと
  署名設定が必須**(エディタとして Xcode を開く必要はほぼないが、初回の Signing 設定と
  Podfile 関連のトラブル時には Xcode でワークスペースを開く)

## 6. 参考: 不採用にした代替アーキテクチャ

| 案 | 不採用理由 |
|----|-----------|
| 全編ネイティブ(Swift/Kotlin 二重開発) | UI・ロジックの二重実装。制約(可能な限り Flutter)に反する |
| `flutter_gl` / ANGLE 系プラグインで Dart から GL | メンテ状況が不安定。ネイティブ直書きより信頼性が低い |
| WebGL + WebView | カメラアクセス・性能・体験すべて劣る |
| 撮影後フィルタのみ(プレビューは素通し) | 「スライダーでプレビューが変わる」というコア体験が失われる |
| ネイティブ通信の FFI 化(MethodChannel の代替) | 下記 §6.1 参照。速度向上がほぼゼロで実装コストのみ増える |

### 6.1 FFI 不採用の検討記録(2026-07-02)

「速度向上前提で MethodChannel を dart:ffi で代用できるか」を検討し、**不採用**とした。

- **速度向上がない**: 重いデータ(カメラフレーム ~2.7MB×30fps)はそもそも Dart を
  経由しない設計(§1 案D)。チャンネルを通るのは制御プレーンのみで、最大でも
  `setFilterParams` の 20 double ≒ 160 byte × 60Hz。MethodChannel のレイテンシ
  (数十 µs〜1ms)はフレーム予算 33ms に対して 3 桁小さく、FFI 化しても体感・計測の
  いずれも差が出ない。
- **実装コストが大きい**: FFI は C ABI しか呼べず、AVFoundation(Swift/ObjC)・
  CameraX(Kotlin)に到達するには iOS は C シム、Android は JNI ブリッジ
  (Dart→FFI→C→JNI→Kotlin の多段)が必要。スレッド安全性・メモリ管理・例外伝播を
  自前設計することになり、「低レベル LLM が確実に実装できる」方針と衝突する。
  ffigen/jnigen による直接 interop も成熟途上で、採用基準(知名度・安定性)を満たさない。
- **将来の限定的な再検討条件**: P0 実装後のプロファイリングで `setFilterParams` の
  反映ジッター(スライダー追従のカクつき)が実測された場合のみ、次の順で対処する。
  1. MethodChannel の範囲内で対処(Android: `TaskQueue` によるバックグラウンド受信)
  2. それでも不足する場合に限り、**FilterParams 専用の FFI ファストパス**
     (C 関数 1 本で 20 float を共有メモリへ書き、レンダースレッドが毎フレーム読む)を追加
  現設計はネイティブ側が atomic 参照差し替え+毎フレーム最新値読みでありジッター耐性は
  高いため、2 まで必要になる可能性は低い。
